# Model Validation

你提到后续可能会用不同大模型做交叉验证。这个文件给出一套实际可执行的方法，避免“每个模型都问一遍，但没有汇总标准”。

## 1. 交叉验证的目标

建议把验证分成三类，而不是混在一起：

- 文案质量：是否清楚、可信、有招生转化能力
- 教学设计：两天议程是否可执行，20 个场景是否适合现场教学
- 交付风险：工作台、账号、网络、招生、审批流程有没有明显漏洞

## 2. 推荐分工

- 模型 A：检查招生文案结构、信息缺漏、表达说服力
- 模型 B：检查教学设计是否过载、节奏是否合理、成果是否可交付
- 模型 C：检查技术与运营风险，包括工作台、token、报名、合规和现场执行

## 3. 推荐提问模板

### Prompt 1：验证招生简章

```text
你是一名高校继续教育项目审核人。请从招生转化、信息完整性、可信度、风险提示四个维度，审查这份 AI+体育应用开发实战工作室招生简章。请输出：
1. 必须修改的问题
2. 建议优化的问题
3. 还缺失的关键信息
4. 是否适合正式公开发布，以及理由
```

### Prompt 2：验证两天议程

```text
你是一名成人教育课程设计顾问。请评估这份两天 AI+体育应用开发实战工作室议程是否适合现场交付。请重点看：
1. 每个模块是否过载
2. 学员是否真的有时间做出成果
3. 哪些环节最容易拖堂或失控
4. 需要增加哪些助教、预习或材料支持
```

### Prompt 3：验证技术与运营风险

```text
你是一名培训项目交付经理。请审查这套 AI 工作台和招生执行方案，找出最可能导致现场失败的前 10 个风险，并按严重程度排序。请同时给出对应的预防动作和兜底方案。
```

## 4. 输出整理方法

每次让不同模型输出后，不要直接改网站，先做一个三列汇总：

| 问题 | 是否多模型共识 | 处理动作 |
| --- | --- | --- |
| 缺少报名联系人 | 是 | 立即补齐 |
| 议程过满 | 是 | 重新压缩 Day 1 下午 |
| 场景数量太多 | 否 | 由总教练决定是否保留 20 个模板但只重点演示 8 到 10 个 |

建议规则：

- 两个及以上模型都指出的问题，优先处理
- 只有一个模型指出的问题，先看是否有现场证据或业务依据
- 任何涉及审批、收费、联系人、开票、网络、账号的风险，都按高优先级处理

## 5. 何时停止验证

出现以下状态时，可以停止大规模交叉验证，进入定稿：

- 不再出现新的高风险问题
- 争议点只剩风格差异，而非结构性缺漏
- 工作台演练已经可以跑通
- 项目负责人和总教练对外口径一致

## 6. 推荐执行顺序

1. 先让模型审网站文案。
2. 再让模型审两天议程和 20 个场景。
3. 最后让模型专门审技术和运营风险。
4. 汇总后只改一轮核心问题，不要无限迭代。
